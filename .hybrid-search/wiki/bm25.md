# Bm25
> synthesized: 2026-04-14

## Overview

The BM25 module provides full-text keyword search as one leg of the hybrid search pipeline. It wraps the Tantivy Rust search engine (via `tantivy-py`) to index code chunks across multiple text fields -- symbol names, qualified names, source content, and docstrings -- so that exact keyword matches complement the semantic vector search provided by [[embedder----openai-api-backend]]. Without this module, queries containing precise identifiers or API names would rely solely on embedding similarity, which often under-ranks exact lexical matches.

## Key Design Decisions

- **`chunk_id` uses the `raw` tokenizer**: The `chunk_id` field is stored with `tokenizer_name="raw"` to prevent any tokenization, ensuring it functions purely as a lookup key rather than a searchable text field (`src/hybrid_search/search/bm25.py:L63`).
- **`content` and `docstring` are not stored**: These fields have `stored=False`, meaning they are indexed for search but their full text is not retrievable from Tantivy -- the authoritative content lives in SQLite instead, avoiding data duplication (`src/hybrid_search/search/bm25.py:L66-L67`).
- **Schema mismatch triggers automatic recreation**: In write mode, if the existing index schema doesn't match the expected schema, the entire index directory is deleted and rebuilt rather than attempting migration. This is safe because the BM25 index is a derived artifact that can be reconstructed from SQLite data (`src/hybrid_search/search/bm25.py:L101-L106`).
- **Over-fetching is absent in BM25 (unlike VectorEngine)**: The `search` method queries Tantivy with exactly the requested `limit`, without the 3x over-fetch strategy used by [[storage-layer]]'s VectorEngine for post-filter headroom ().
- **Two-pass query escaping**: If the initial `parse_query` fails (e.g., due to special characters in user input), the query is escaped with `_escape_tantivy_query` and retried. If that also fails, an empty result is returned rather than raising an exception (-L180`).

## Caveats

- **Lazy writer allocation**: The `_writer` is created on first mutation via `_get_writer()` with a fixed 50MB heap. If many documents are added without calling `commit()`, memory usage can grow unboundedly within that heap (`src/hybrid_search/search/bm25.py:L120-L123`).
- **`delete_batch` is not atomic**: Each deletion calls `delete()` individually in a loop. If an error occurs mid-batch, some documents may be deleted while others remain (`src/hybrid_search/search/bm25.py:L149-L152`).
- **`read_only` mode silently returns empty results**: When the index cannot be opened in read-only mode, `_index` is set to `None` and all searches return `[]` with only a warning log. Callers have no way to distinguish "no results" from "index unavailable" (`src/hybrid_search/search/bm25.py:L96-L100`).
- **`count` property requires reload**: The `count` property calls `self._index.reload()` on every access, which may have performance implications if called frequently (-L203`).
