# v0.3.0 — Deterministic Memory Layer

**Date**: 2026-04-23
**Author**: drafting session (to be executed autonomously)
**Status**: plan draft → ready to execute

---

## Problem statement

v0.2.0 shipped with the "compounding quality" claim backed by a benchmark.
Real-world use with valuein_homepage exposed two design leaks that make the
claim unreliable in practice:

1. **qa_log save rate is stochastic** — persistence fires only when Claude
   chooses the `hybrid_search` MCP tool. In observed sessions, exploratory
   Korean NL queries triggered MCP ~60% of the time; the remainder went to
   Grep/Read and produced zero memory entries. Same-category questions
   produced wildly different save outcomes turn-to-turn.

2. **Wiki accumulates orphans** — audit of valuein shows **765 / 2,584 pages
   (29.6%) are zombies**: they reference file paths that are no longer in
   the index (gitignore drift or actual deletion). The reindex pipeline
   currently has no "delete wiki page when its sources disappear" step.

Together these mean:
- Users think memory accumulates "automatically" but roughly half of
  exploratory turns are silently dropped.
- Over months, the wiki becomes bloated with broken references that still
  surface in `hybrid_search` results as noise.

v0.3.0 closes both leaks so the product promise holds deterministically.

## Architecture changes

### New hook lifecycle (per turn)

```
 User submits prompt
   ↓ UserPromptSubmit hook fires (NEW)
   |   1. Classify prompt (exploratory vs precision vs command)
   |   2. If exploratory: run hybrid_search programmatically
   |   3. Inject top-10 as additionalContext JSON
   ↓
 Claude processes with enriched context
   ↓ Existing PreToolUse hook still fires on Grep/Read (nudge only)
   ↓
 Claude finishes answering
   ↓ Stop hook fires (NEW)
   |   1. Read transcript, extract last user-assistant pair
   |   2. Save qa_log entry tagged trigger="stop_hook"
   |   3. Include tools-used list + answer excerpt
```

**Key property**: UserPromptSubmit owns *retrieval*, Stop owns *save*. The
MCP tool's existing save path stays for backward compat but is no longer
load-bearing — Stop catches everything regardless of Claude's tool choice.

### Classification heuristics (UserPromptSubmit)

A prompt is **exploratory** when *any* of:
- Contains question words (어떤, 어떻게, 왜, 뭐, what, how, why, explain, 설명, 정리)
- Korean NL fragment of length ≥ 10 chars
- Contains memory-intent markers (지난번, 이전에, previously, earlier)

Skip classification when:
- Starts with `/` (slash command)
- Starts with `!` (bash passthrough)
- Starts with `@` (mention / file reference)
- Single-word prompt shorter than 15 chars (too noisy to be worth enriching)

Heuristics are cheap — no LLM involvement. False positives cost one extra
search (~300ms, ~5K tokens injected). False negatives silently fall back
to v0.2.0 behaviour.

### qa_log format v2

Backward-compatible additions to the existing frontmatter:

```yaml
---
# v1 fields (unchanged)
query: "..."
query_type: ...
effective_bm25_weight: 0.4
query_time_ms: 448.0
total_chunks_searched: 23558
timestamp: ...
result_count: 10
# v2 fields (new, optional)
trigger: "stop_hook" | "mcp_tool" | "user_prompt_submit"
tools_used: ["Grep", "Read", "mcp__hybrid-search"]
answer_chars: 1234
---
```

Body format unchanged. `tools_used` enables future analytics ("how often
does memory actually help vs when is Claude still bypassing it").

### Wiki orphan detection (DB-based)

Replaces the current disk-based liveness check with a query against the
store DB:

```python
def is_wiki_orphan(wiki_path: Path, db: StoreDB) -> bool:
    refs = extract_file_refs(wiki_path.read_text())
    if not refs:
        return False  # empty pages (index) are kept
    indexed_paths = {row["relative_path"] for row in db.all_files()}
    return all(ref.split(":", 1)[0] not in indexed_paths for ref in refs)
```

Fires at end of reindex. Also callable standalone via
`hybrid-search-mcp wiki-cleanup --cwd .`.

## Risk register

| Risk | Mitigation |
|---|---|
| Stop hook input format not as expected | Verify before coding via claude-code-guide agent; fall back to minimal save (question + timestamp) if transcript parsing fails |
| UserPromptSubmit latency hurts UX | Cap `hybrid_search` call to 500ms with hard timeout; silent failure returns empty context — user doesn't see delay |
| Double-save between existing MCP path and new Stop hook | Use fingerprint (query+timestamp) to dedupe; Stop hook checks if same query was saved < 5s ago |
| Transcript file not readable by hook | Fall back to question-only save; log at debug |
| New hook types unsupported by installed Claude Code version | Detect at `install-memory-hook` time; warn user if version < 2.1.9 |
| Bench re-run shows regression | Investigate before ship; if valid tradeoff, document it; if bug, block release |
| Wiki cleanup removes something it shouldn't | DB-based detection is strictly stricter than disk-based; plus dry-run default + explicit confirm |

## Rollback plan

Each stage is committed separately. If final verification fails on a
specific goal, revert the last failing stage's commits and ship what's
green (versioned as `0.3.0-rc1` with known limitations documented).

Goals G1, G2, G3 are the release-critical ones. G4/G5 gate testing
hygiene. G7/G8/G9 are polish — missing them is a warning, not a blocker.
