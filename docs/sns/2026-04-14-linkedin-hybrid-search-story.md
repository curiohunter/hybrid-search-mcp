# LinkedIn 게시글 — Hybrid Search MCP 개발기

바이브 코딩을 하다 보면 한국어로 묻게 됩니다.

"숙제분석 기능 보여줘"
"학원비 세션 연동 어떻게 돼?"

그런데 코드베이스는 영어입니다.
`HomeworkAnalysisTab`, `saveTuitionFeesWithSessions` 같은 이름을 미리 알아야 Grep이 제대로 먹히죠.

그래서 한국어 자연어 질문으로 영어 코드베이스를 찾을 수 있는 hybrid search + wiki 도구를 직접 만들었습니다.

---

처음엔 Karpathy의 `llm-wiki` 류 도구들이 흥미로워서, `llm-wiki-compiler`, `mindvault` 같은 걸 실제 프로젝트에 붙여봤습니다.

1,700파일짜리 Next.js 프로젝트에 적용해보니 제 용도에는 조금 아쉬웠습니다. 프로젝트 규모에 비해 wiki가 압축적으로 생성돼서, 제가 보고 싶은 모듈별 구조와 상세 흐름까지는 잘 안 내려오더라고요.

그래서 "그럼 내 용도에 맞게 직접 만들자"로 방향을 틀었습니다.

핵심은 이렇습니다.

- AST로 코드/문서를 함수, 클래스 단위로 청킹
- BM25 + Vector 하이브리드 검색
- Call Graph로 호출 관계 연결
- 그 위에 구조적 wiki 자동 생성

특히 중요했던 건 "한국어 질문 ↔ 영어 코드"를 이어주는 부분이었습니다.

예를 들어 저는 "숙제분석 기능 알려줘"라고 묻습니다.
코드는 `HomeworkTab`, `StudentAnalysisTab`, `useStudentAnalysis` 같은 영어 이름으로 되어 있고요.

Grep은 정확한 키워드를 알아야 강합니다.
반면 벡터 검색은 "숙제분석"이라는 한국어 질의를 영어 코드와 의미적으로 연결해줍니다.

---

임베딩 쪽은 꽤 삽질했습니다.

처음엔 로컬로 끝내고 싶어서 Ollama에 Qwen3-Embedding 0.6B를 올렸습니다. 맥북 하나로 개발하는 1인 개발자 입장에서는 로컬 임베딩이 매력적이었거든요.

벤치마크도 직접 돌려봤고, 한국어 Recall@1도 0.70 정도로 나쁘지 않았습니다.

그런데 실제로 1,700파일을 인덱싱하니 맥북이 거의 인덱싱 머신이 되어버렸습니다. CPU도 태워보고, GPU(MPS)도 써보고, 배치도 줄여봤는데 결국 작업 머신으로 쓰기가 어려웠습니다.

그래서 OpenAI `text-embedding-3-small`로 갈아탔습니다.
전체 인덱싱 비용이 약 $0.04.
지금 생각하면 처음부터 이걸 쓸 걸 그랬습니다.

---

지금은 이렇게 씁니다.

"학원비 세션 연동 알려줘"
→ 검색 1회, 약 3초
→ 서비스 코드, DB 스키마 문서, 관련 흐름까지 같이 나옴

직접 써보니 Grep + Read 반복보다 훨씬 편했습니다.
도구 호출 수는 줄고, 전체 흐름 파악 속도는 훨씬 빨라졌습니다.

특히 wiki가 "도메인 전체 그림"을 잡아주고, hybrid search가 세부 코드와 문서를 채워주는 조합이 꽤 괜찮았습니다.

아직은 제 프로젝트들에서 계속 검증 중입니다.
실사용 사례를 더 쌓아봐야 하고, 자동 인덱싱이나 wiki 갱신도 더 다듬어야 합니다.

그래도 오랜만에 "벡터 검색이 실제로 쓸모 있다"는 감각을 다시 얻은 프로젝트였습니다.

완성품이라기보다는, 제가 진짜 필요해서 만든 개발 도구에 가깝습니다.
